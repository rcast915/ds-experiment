The compiler transformation pass approach predicted at the end of ds_results.txt has been implemented. A PJRT proxy plugin intercepts JAX compile calls, runs an MLIR pass (ds-transform) over the StableHLO bytecode, and returns modified bytecode to the real CUDA backend. The pass expands float arithmetic into double-single sequences as native StableHLO ops, so XLA sees and can fuse the full expanded sequence with no custom-call boundaries. This directly addresses the fusion limitation identified in the FFI experiments.

The inline compiler pass avoids the steep arithmetic-intensity slowdown observed with separate FFI custom calls. Because the expanded DS ops are native StableHLO, XLA schedules and register-allocates across the entire kernel rather than treating each DS operation as an opaque boundary. FMA contraction is not a risk: inspection of XLA-dumped PTX on the H100 shows zero fma.rn.f32 instructions in the Veltkamp split sequence, confirming that per-instruction IEEE .rn rounding prevents contraction at the PTX assembler level.

The compiler pass correctly handles float64 inputs as well as float32. For f64 inputs, function arguments are split at function entry into (f32_hi, f32_lo) pairs via exact arithmetic (hi = float32(x), lo = float32(x - float64(hi))), all intermediate computation is performed in f32 DS arithmetic, and the result is recombined to f64 at function exit. The caller observes no type change and the transformation is fully transparent.

On the H100, DS-f32 runs at approximately the same wall time as native f64 across all tested operation types. For element-wise operations (A*A+B) and reductions (jnp.sum), DS-f32 runs within 10% of native f64 speed at sizes from 1K to 1M elements. For matrix multiply at 2048x2048, DS-f32 is 1.13x faster than f64; at smaller sizes it is modestly slower (0.72x at 256x256 due to four kernel launches versus one). Neither baseline shows a large performance gap on this hardware.

The reason DS-f32 and f64 perform comparably on H100 is that the H100 is a scientific HPC GPU with native FP64 tensor core support (~51 TFLOPS on the PCIe variant). Both f64 and DS-f32 are therefore operating at similar levels of hardware utilization for these workloads. On consumer GPUs and other hardware without dedicated FP64 units, f64 throughput is typically 1/32 to 1/64 of f32 throughput, and DS-f32 would run dramatically faster than f64 while preserving near-f64 accuracy.

The practical claim supported by these results is therefore not that DS-f32 beats f64 universally, but that DS-f32 achieves near-f64 numerical accuracy at approximately f32 hardware cost. On systems with poor f64 support this is a direct speedup; on systems with good f64 support such as H100 it means f64-class accuracy with no performance penalty relative to f64 and at lower memory bandwidth cost (two f32 values versus one f64 value has the same memory footprint but uses f32 arithmetic paths throughout).

The precision benefit of DS-f32 over naive f32 is confirmed with f64 inputs. For a reduction over 10,000 elements of 0.1 (f64 input), DS-f32 produces an error of 1.49e-05 versus f32 error of 1.22e-04, an 8x improvement. This is consistent with DS-f32's effective ~48-bit mantissa versus f32's 24 bits, and matches the numerical improvements observed in the original FFI-based GPU experiments. The DS-f32 error remains well below f32 error even as problem size grows, confirming that the lo channel correctly accumulates and carries rounding residuals through the compiler-transformed kernels.
